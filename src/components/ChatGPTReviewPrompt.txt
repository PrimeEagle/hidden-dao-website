🔧 React + TypeScript Project Review Guidelines (Vite + Tailwind + TanStack Start)
📂 1. Structure & Organization

Each component lives in its own folder:

src/components/[ComponentName]/


Within that folder:

components/   → visual subcomponents only  
hooks/        → internal logic or lifecycle  
behaviors/    → optional or variant logic (animations, autoplay, transitions, etc.)
  [behaviorGroup]/  → group folder for related variants  
    fade.ts
    slide.ts
    zoom.ts
    shift.ts
    index.ts   → auto-generated by Barrelsby (named exports only)
utils/        → pure stateless helpers  
types/        → all TypeScript definitions  
index.ts      → barrel export (auto-generated)


All barrels are generated automatically by Barrelsby and export named symbols only (no default exports).

All imports must use alias paths (@/components/...) instead of relative paths.

🧩 2. Naming Conventions
Type	Convention	Example
Components (.tsx)	PascalCase	ContentSlider.tsx, NavButton.tsx
Hooks (.ts)	use[ComponentName][Purpose]	useContentSliderController.ts
Behaviors (.ts)	camelCase	fade.ts, slide.ts, zoom.ts, loop.ts
Utilities (.ts)	camelCase	calculateOffset.ts, clampIndex.ts
Type definitions (.ts)	[componentName].types.ts	hoverPanel.types.ts
Barrels	index.ts (auto-generated)	named exports only

Each behavior variant file exports a single variant function or object.
Barrels automatically re-export all variants with named exports.

🎨 3. Styling & Presentation

TailwindCSS only — no external .css or .scss files.

Layouts and animations must be responsive, mobile-first, and accessible.

Components support flexible visual composition via children or render props.

⚙️ 4. Functional Goals

Refactors must preserve all existing functionality.
Extract reusable logic into hooks, behaviors, or utilities.

Ensure:

✅ Accessibility (ARIA attributes, keyboard nav, focus rings)

✅ SSR compatibility (no DOM access outside effects)

✅ Responsiveness via Tailwind breakpoints

🧠 5. Behavioral Architecture

Required logic → hooks/ (e.g., useContentSliderController.ts)

Optional or variant logic → behaviors/ grouped by type

Utilities → stateless helpers shared across all layers

Each behavior group (animations/, autoplay/, transitions/, etc.) contains:

multiple variant files

a Barrelsby-generated index.ts with named re-exports only

🔄 6. Variant Selection

Behavior groups no longer export runtime strategy objects.

Instead, import all variants as a namespace using the * as syntax:

import * as animationStrategies from "@/components/HoverPanel/behaviors/animations";
import * as autoplayStrategies from "@/components/HoverPanel/behaviors/autoplay";
import * as expansionStrategies from "@/components/HoverPanel/behaviors/expansion";


Then define variant types from the namespace:

type AnimationType = keyof typeof animationStrategies;
type ExpansionType = keyof typeof expansionStrategies;


Use those keys to select variants:

const s = expansionStrategies[type];
s.isExpanded(...);

✅ Supported usages

Static

<HoverPanel animationType="fade" />


Dynamic

const [animation, setAnimation] = useState("slide");
<HoverPanel animationType={animation} />;


Custom

<HoverPanel
  animationType="fade"
  customAnimation={(el, expanded) => {
    el.style.transition = "all 400ms ease";
    el.style.transform = expanded ? "rotate(0)" : "rotate(-3deg)";
  }}
/>


This pattern ensures type safety, no manual mapping objects, and automatic variant discovery from barrels.

⚖️ 7. Path Aliases

Use alias-based imports for clarity and refactor safety:

import { PanelItem } from "@/components/HoverPanel/components";


Configured in tsconfig.json:

"paths": {
  "@/*": ["src/*"]
}

🧱 8. Core Principles

Single responsibility per file

Named exports only (no default)

Tailwind-only styling

SSR-safe and accessible

Behavior modularity via per-file variants

Variant injection through type-safe keys

Always import behavior groups via * as namespace (e.g. import * as animationStrategies ...)

🧾 9. Refactor Review Process

When reviewing a component:

Identify its subcomponents, hooks, behaviors, utilities, and types.

Show resulting files in full (new files, updated imports/exports, barrels).

Confirm Barrelsby integration and alias imports resolve cleanly.

Verify identical runtime behavior.

Summarize:

Structural rationale

Accessibility & SSR improvements

Responsiveness adjustments
