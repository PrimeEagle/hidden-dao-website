You are reviewing a React + TypeScript project built with Vite, Tailwind, and TanStack Start.

My goals are:

📂 1. Structure & Organization

Each component lives in its own folder:
src/components/[ComponentName]/

Within that folder, use the following substructure when applicable:

components/   → visual subcomponents only  
hooks/        → required internal hooks for logic or lifecycle  
behaviors/    → optional or variant logic (e.g., animations, autoplay, transitions)
  [behaviorGroup]/  → folder for related variants (e.g., animations/, transitions/, autoplay/)
    fade.ts
    slide.ts
    zoom.ts
    shift.ts
    index.ts         → aggregates all variants into a single exported strategies object  
utils/        → pure non-React helpers or shared internal functions  
types/        → all TypeScript definitions  
index.ts      → barrel export (named exports only)


Use Barrelsby for automatic index.ts generation, and export only named exports (never default).

Maintain path alias imports using @/components/... rather than relative paths.

🧩 2. Naming Conventions

Components (.tsx) → PascalCase
Example: ContentSlider.tsx, NavButton.tsx

Hooks (.ts) → use[ComponentName][Purpose].ts
Examples:

Required logic: useContentSliderController.ts

Optional or variant logic: useContentSliderFade.ts, useContentSliderCube.ts

Behavior Variants (.ts) → camelCase filenames
Examples:

fade.ts, slide.ts, zoom.ts, shift.ts, loop.ts

Each file exports a single variant function or object.

Each group has an aggregator file (e.g., behaviors/animations/index.ts or behaviors/autoplay/index.ts)
that imports all variants and creates a strategy map, such as:

export const animationStrategies = { fade, slide, zoom, shift }
export const autoplayStrategies = { linear, pulse, reverse }


Utilities (.ts) → camelCase filenames describing functionality
Example: calculateOffset.ts, clampIndex.ts

Type definitions (.ts) → [componentName].types.ts

Index files (.ts) → index.ts, exporting only named exports

🎨 3. Styling & Presentation

Use TailwindCSS only — no external .css or .scss files.

Layout and animation styling must be responsive, mobile-first, and accessible.

Support visual flexibility via render props or children for external customization.

⚙️ 4. Functional Goals

Preserve all functionality — refactors must not alter behavior.

Extract reusable logic into hooks, behaviors, or utilities.

Ensure:

Accessibility (A11y) — include aria-* attributes, keyboard navigation, and visible focus states.

SSR Compatibility — avoid direct DOM references outside effects or dynamic imports.

Responsiveness — handle layout shifts using Tailwind breakpoints.

🧠 5. Behavioral Architecture

Required logic lives in hooks/ (e.g., useContentSliderController.ts).

Optional or variant logic lives in behaviors/, grouped by type.

Each behavior group (e.g., animations/, autoplay/, transitions/) contains multiple variant files and an index.ts that aggregates them.

Behaviors can be:

Imported conditionally inside components, or

Injected via props (e.g., behaviorHook or strategyType).

Utilities remain stateless, reusable helpers shared across logic layers.

This pattern clarifies which logic is required, and which is variant, optional, or swappable.

🔄 6. Variant Selection

Each behavior group exports a unified strategies map.
For example, animationStrategies, autoplayStrategies, or transitionStrategies.

You can select variants in three ways:

A. Static (default)

Define behavior via prop:

<HoverPanel animationType="fade" />

B. Dynamic (runtime)

Switch behavior dynamically:

const [animation, setAnimation] = useState("slide")
<HoverPanel animationType={animation} />

C. Custom (injected)

Inject a custom variant directly:

<HoverPanel
  animationType="fade"
  customAnimation={(el, expanded) => {
    el.style.transition = "all 400ms ease";
    el.style.transform = expanded ? "rotate(0)" : "rotate(-3deg)";
  }}
/>


The controller hook should support both string keys and injected functions, e.g.:

export function usePanelBehavior<T extends keyof typeof animationStrategies>(
  type: T,
  custom?: (el: HTMLElement, active: boolean) => void
)

⚖️ 7. Path Aliases

Use alias-based imports for clarity and refactor safety:

import { PanelItem } from "@/components/HoverPanel/components"


In tsconfig.json:

"paths": {
  "@/*": ["src/*"]
}

🧱 8. Core Principles

Single responsibility per file

Named exports only — no default exports

Tailwind-only styling

SSR-safe and accessible

Behavioral modularity — every interchangeable behavior lives in its own file, with an aggregator index

Variant injection — components can receive behaviors as props or use built-in strategy maps
